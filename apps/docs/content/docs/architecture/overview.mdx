Veritio uses a split frontend/backend architecture with event-driven processing. The frontend is a Next.js application, the backend runs on the Motia framework powered by the iii engine, and they communicate via HTTP proxying and WebSocket connections.

## Port Map

| Port | Service | Protocol | Purpose |
|------|---------|----------|---------|
| **4000** | iii engine | HTTP | API routing, health check, all backend endpoints |
| **4001** | Next.js | HTTP | Frontend UI, SSR, auth routes (`/api/auth/*`) |
| **4002** | Yjs server | WebSocket | Real-time collaborative editing |
| **4004** | Stream module | WebSocket | Live data streaming (assistant chat, real-time updates) |
| **6379** | Redis | TCP | Job queues, state storage, stream pub/sub |
| **49134** | iii engine (internal) | WebSocket | Step handler to engine communication (not externally exposed) |

## Request Flow

When the browser makes an API call, the request follows this path:

1. **Browser** makes an API request to `http://localhost:4001/api/studies/123`.
2. **Next.js** rewrites the request to `http://localhost:4000/api/studies/123` via rewrite rules configured in `next.config.ts`.
3. **iii engine** receives the request and matches it to a registered step handler based on the HTTP method and path.
4. **Auth middleware** validates the session token by checking the `Authorization: Bearer` header or session cookie against the `session` table in PostgreSQL.
5. **Step handler** processes the request: queries PostgreSQL, reads or writes Redis state, and optionally enqueues events for asynchronous processing.
6. The response flows back through the same path to the browser.

### Auth Route Exception

Auth routes (`/api/auth/*`) are **not** proxied to the iii engine. They are handled directly by Better Auth within the Next.js process. This includes sign-in, sign-up, session management, and OAuth callbacks.

There is one exception to this exception: Figma auth routes (`/api/auth/figma/*`) **do** get proxied to the Motia backend because Figma OAuth integration is handled by a step handler, not by Better Auth.

## Monorepo Structure

The project is a Turborepo + Bun monorepo:

- `apps/veritio/` -- Main application (Next.js frontend + Motia backend)
- `apps/docs/` -- Documentation site (Fumadocs)
- `packages/@veritio/*` -- 11 shared packages (UI, auth, core, study types, study flow, card sort, dashboard common, analysis shared, SWR config, Yjs, prototype test)
- `workers/` -- Cloudflare Worker for reverse-proxy mode in live website testing

## Frontend: Next.js App Router

Route groups in `src/app/`:

| Group | Purpose |
|-------|---------|
| `(dashboard)/` | Authenticated dashboard -- projects, studies, builder, results |
| `(participant)/` | Participant-facing study player (`/s/[studyCode]`) |
| `(auth)/` | Sign-in, sign-up, invite flows |
| `(admin)/` | Superadmin panel (guarded by `SUPERADMIN_USER_ID`) |
| `(public)/` | Public routes -- shared results, widget preview |
| `render/pdf/` | Server-rendered pages for PDF export via Puppeteer |

### API Proxy Configuration

In `next.config.ts`, the rewrite rules route API traffic:

```
/api/* (except /api/auth/*) --> rewrites to http://localhost:4000/api/*
```

The frontend proxies all API calls through the iii engine, except auth routes which Better Auth handles directly inside Next.js.

### Image Sources

Allowed remote image sources: Supabase storage (`*.supabase.co/storage/v1/object/public/**`) and Unsplash (`images.unsplash.com`).

### Cache Headers

- Static assets (`/images/*`, `*.svg`): `max-age=31536000, immutable` (1 year)
- Participant pages (`/s/*`): `max-age=0, must-revalidate` (never cached, ensures fresh study data)

## Backend: Motia Steps

API endpoints are Motia "steps" in `apps/veritio/src/steps/`. Files must end with `.step.ts` for auto-discovery by the Motia compiler.

| Type | Directory | Purpose |
|------|-----------|---------|
| API Steps | `src/steps/api/` | HTTP endpoint handlers (~50+ feature domains) |
| Event Steps | `src/steps/events/` | Async event processors (~15 handlers via BullMQ) |
| Cron Steps | `src/steps/cron/` | Scheduled tasks (~10 jobs) |
| Stream Steps | `src/steps/streams/` | Real-time WebSocket handlers |

Each step exports a `config` object (defining triggers, name, and enqueued events) and a `handler` function. Business logic typically lives in `src/services/`.

For detailed coverage of step types, API domains, cron jobs, and queue event flow, see the [Backend](/docs/architecture/backend) page.
